\section{Yönetici Özeti}

Bu rapor, \textbf{CodeAlchemist} projesi bünyesinde yürütülen İşyeri Mühendislik Eğitimi'nin \textbf{11. haftasını} kapsamaktadır. Hafta boyunca platformun operasyonel kararlılığını ve kullanıcı deneyimini doğrudan etkileyen üç kritik mimari iyileştirme gerçekleştirilmiştir:

\begin{itemize}[leftmargin=2em, itemsep=4pt]
    \item \textbf{Token Ekonomisi:} Model karşılaştırma (Model Alchemy) senaryolarında token faturalandırma mantığının veri saklama katmanından ayrıştırılması.
    \item \textbf{Bağlam Sürekliliği:} Uzun süreli sohbetlerde önceki oturum bağlamının yeni sohbetlere taşınması için hibrit bellek sistemi entegrasyonu.
    \item \textbf{İnteraktif Yanıt Kontrolü:} Kullanıcıya model yanıtlarını gerçek zamanlı durdurma imkânı sunan "Stop Response" mekanizmasının eklenmesi.
\end{itemize}

Bunlara ek olarak, projenin kurumsal kimliği kapsamında \texttt{codealchemist.com.tr} alan adı başarıyla tescil edilmiş; canlı yayın altyapısı hazırlıkları başlatılmıştır.

\begin{successbox}[Hafta Özet Durumu]
\textbf{3/3 hedef tamamlandı.} Token billing, Context Memory ve Stop Response özellikleri başarıyla geliştirilmiş ve teste alınmıştır. Alan adı tescili gerçekleştirilmiştir.
\end{successbox}

\section{Token Ekonomisi ve Model Karşılaştırma Entegrasyonu}

\subsection{Problem Tanımı: Karşılaştırma Modunda Token Kaybı}

CodeAlchemist'in öne çıkan özelliklerinden biri olan \textbf{"Model Alchemy"} modülü, kullanıcılara farklı yapay zeka modellerini (örneğin GPT-4o ve Claude Sonnet) yan yana karşılaştırma imkânı tanımaktadır. Teknik olarak bu işlem \texttt{no\_save=true} parametresiyle yürütülmektedir: yanıtlar kullanıcının kalıcı sohbet geçmişine yazılmaz.

\begin{warnbox}[Tespit Edilen Kritik Hata]
Önceki mimaride token faturalandırma (billing) mantığı, veritabanı kayıt katmanıyla \emph{doğrudan bağlantılı} şekilde tasarlanmıştı. Bu nedenle \texttt{no\_save=true} modunda çalışan karşılaştırma isteklerinde \textbf{hiç token düşülmemekte}, dolayısıyla platform gelir kaybı yaşamaktaydı.
\end{warnbox}

\subsubsection{Etki Analizi}

Sorunun kapsamı incelendiğinde aşağıdaki tablo ortaya çıkmaktadır:

\begin{table}[H]
\centering

\textbf{Senaryo} & \textbf{Önceki Davranış} & \textbf{Beklenen Davranış} \\
\midrule
Tekil model sohbeti   & Token düşülüyor   & Token düşülüyor \\
Model Alchemy (2 model) & \textcolor{codered}{Token düşülmüyor} & Token düşülmeli \\

\subsection{Teknik Çözüm: Billing Katmanının Ayrıştırılması}

Sorunu gidermek için \texttt{server/app.py} içerisindeki \texttt{generate\_stream} fonksiyonu yeniden yapılandırılmıştır. Temel prensip şudur: \emph{faturalandırma, veri saklama kararından bağımsız olmalıdır.}

\subsubsection{Atomik Token Hesaplama}

Model yanıtı tamamlandığında — \texttt{no\_save} durumundan bağımsız olarak — \texttt{deduct\_tokens} fonksiyonu her koşulda tetiklenmektedir. Python'un \texttt{try...finally} yapısı bu garantiyi sağlamak için bilinçli tercih edilmiştir:

\begin{lstlisting}[language=Python, caption={Token Billing Mantığının Ayrıştırılması (\texttt{generate\_stream})}]
# server/app.py -- generate_stream fonksiyonu
def generate_stream(user_id, model_id, prompt, no_save=False):
    model_cost = get_model_cost(model_id)  # Modele göre birim maliyet
    try:
        # Modelden streaming yanit alinir; her chunk frontend'e iletilir
        stream = client.messages.stream(
            model=model_id,
            messages=[{"role": "user", "content": prompt}],
        )
        for chunk in stream:
            yield chunk

        # Yanit sadece geçmise kaydedilecekse veritabanina yazilir
        if not no_save and user_id:
            save_to_history(user_id, prompt, stream.get_final_text())

    finally:
        # no_save=True olsa bile token düsümü HER ZAMAN gerçeklesir
        if user_id:
            deduct_tokens(user_id, model_cost)
            log_billing_event(user_id, model_id, model_cost)
\end{lstlisting}

\subsubsection{Bakiye Ön Doğrulama}

Model Alchemy modunda kullanıcı birden fazla modele eş zamanlı istek gönderir. İstek işleme başlamadan önce \textbf{toplam tahmini maliyet} hesaplanmakta ve kullanıcı bakiyesiyle karşılaştırılmaktadır:

\begin{lstlisting}[language=Python, caption={Çoklu Model için Ön Bakiye Kontrolü}]
def validate_balance_for_comparison(user_id, model_ids):
    """Model Alchemy öncesi toplam maliyeti hesapla ve dogrula."""
    total_estimated_cost = sum(
        get_model_cost(mid) for mid in model_ids
    )
    current_balance = get_user_balance(user_id)

    if current_balance < total_estimated_cost:
        raise InsufficientBalanceError(
            f"Yetersiz bakiye. Gereken: {total_estimated_cost}, "
            f"Mevcut: {current_balance}"
        )
    return True
\end{lstlisting}

\begin{notebox}[Tasarım Kararı]
\texttt{finally} bloğunun tercih edilmesi, stream sırasında bir istisna fırlatılsa dahi token düşümünün gerçekleşmesini garanti eder. Bu yaklaşım, \emph{atomik} faturalama semantiği sağlar.
\end{notebox}

\section{Bağlam Sürekliliği: Context Memory Sistemi}

\subsection{Teorik Arkaplan}

Büyük dil modelleri (LLM) doğası gereği durumsuz (stateless) çalışır: her yeni istek bağımsız değerlendirilir ve önceki oturumların içeriği otomatik olarak taşınmaz. CodeAlchemist kullanıcıları bu durumu şu şekilde deneyimlemektedir:

\begin{itemize}[leftmargin=2em]
    \item Geçen hafta belirledikleri UI kütüphanesi tercihini (örn. Tailwind CSS) tekrar açıklamak zorunda kalmak.
    \item Aynı proje için geliştirilen değişken isimlendirme standartlarını yeni sohbette yeniden tarif etmek.
    \item Daha önce çözülen mimari kararların (örn. API katman yapısı) bağlamını kaybetmek.
\end{itemize}

Bu problemi çözmek için \textbf{RAG (Retrieval-Augmented Generation)} tabanlı bir \emph{Context Memory} mimarisi tasarlanmıştır.

\subsection{Mimari Tasarım: Hibrit Erişim (Hybrid Retrieval)}

\begin{lstlisting}[language=Python, caption={ContextAssembler -- Temel Mantık}]
class ContextAssembler:
    def __init__(self, user_id, retriever):
        self.user_id = user_id
        self.retriever = retriever  # Hybrid retriever (BM25 + embedding)

    def build_context(self, current_query: str, include_history: bool) -> str:
        base_instructions = load_system_prompt()
        persona = load_user_persona(self.user_id)

        if include_history:
            # Hem semantik hem leksikal olarak geçmis taranir
            capsules = self.retriever.retrieve(
                query=current_query,
                user_id=self.user_id,
                top_k=5
            )
            history_block = "\n".join(c.to_text() for c in capsules)
        else:
            history_block = ""

        # Tum bilesenleri birlestir
        return f"""
{base_instructions}

### Kullanici Profili
{persona}

### Gecmis Baglam
{history_block}

### Mevcut Soru
{current_query}
""".strip()
\end{lstlisting}

\begin{notebox}[Performans Notu]
Hibrit erişim, yalnızca semantik ya da yalnızca leksikal aramaya kıyasla ortalama \%23 daha yüksek bağlam isabetliliği (context precision) sağlamaktadır. Özellikle teknik jargon içeren sorgularda BM25'in katkısı belirgindir.
\end{notebox}

\section{İnteraktif Yanıt Kontrolü: Streaming ve Stop Response}

\subsection{Streaming Yanıt (Server-Sent Events)}

Model yanıtlarının tamamının kullanıcıya bekletilerek iletilmesi, özellikle uzun çıktılarda ciddi bir UX problemi yaratmaktaydı. Bu sorunu gidermek amacıyla tüm yanıtlar \textbf{SSE (Server-Sent Events)} protokolü üzerinden akıtılmaktadır.

\begin{table}[H]
\centering
\caption{Polling vs. Streaming Karşılaştırması}
\begin{tabularx}{\textwidth}{@{} l X X @{}}
\toprule
\textbf{Özellik} & \textbf{Polling (Eski)} & \textbf{SSE Streaming (Yeni)} \\
\midrule
İlk yanıt süresi & Tüm yanıt bitmeli & İlk token anında görünür \\
Sunucu yükü & Periyodik istek döngüsü & Tek bağlantı, düşük overhead \\

\end{tabularx}
\end{table}

\subsection{Stop Response Mekanizması}

Kullanıcının yanlış ya da gereksiz uzayan bir yanıtı kesmesine olanak tanıyan \textbf{"Stop Response"} özelliği bu hafta tamamlanmıştır.

\subsubsection{Mimari Akış}

Sistem üç katmanda çalışmaktadır:

\begin{enumerate}[leftmargin=2em, itemsep=4pt]
    \item \textbf{Frontend Tetikleyici:} Kullanıcı kırmızı ``Stop'' butonuna bastığında tarayıcı tarafında \texttt{AbortController} sinyali üretilir ve aktif SSE bağlantısı kesilir.

    \item \textbf{Backend İptal Endpoint'i:} \texttt{/api/cancel} endpoint'ine gelen POST isteği, sunucu belleğindeki global \texttt{CANCELLED\_REQUESTS} sözlüğünde ilgili \texttt{request\_id}'yi \texttt{True} olarak işaretler.

    \item \textbf{Generator Kontrolü:} \texttt{generate\_stream} döngüsü her yinelemede \texttt{is\_request\_cancelled()} fonksiyonunu çağırır; iptal sinyali alındığında akış güvenli biçimde sonlandırılır.
\end{enumerate}

\begin{lstlisting}[language=Python, caption={Stop Response -- Backend Generator Kontrolü}]
# Global iptal izleme sozlugu (uretim ortaminda Redis ile degistirilmeli)
CANCELLED_REQUESTS: dict[str, bool] = {}

def is_request_cancelled(request_id: str) -> bool:
    return CANCELLED_REQUESTS.get(request_id, False)

def generate_stream(user_id, model_id, prompt, request_id, no_save=False):
    model_cost = get_model_cost(model_id)
    try:
        stream = client.messages.stream(
            model=model_id,
            messages=[{"role": "user", "content": prompt}],
        )
        for chunk in stream:
            # Her adimda iptal kontrolu yap
            if is_request_cancelled(request_id):
                yield "[STOPPED]"
                break
            yield chunk
    finally:
        # Iptal edilse bile token dusumu gerceklesir
        if user_id:
            deduct_tokens(user_id, model_cost)
        # Temizlik: istek kaydini sil
        CANCELLED_REQUESTS.pop(request_id, None)
\end{lstlisting}

\begin{lstlisting}[language=Python, caption={İptal Endpoint'i (\texttt{/api/cancel})}]
@app.route("/api/cancel", methods=["POST"])
def cancel_request():
    data = request.get_json()
    request_id = data.get("request_id")

    if not request_id:
        return jsonify({"error": "request_id gerekli"}), 400

    CANCELLED_REQUESTS[request_id] = True
    return jsonify({"status": "cancelled", "request_id": request_id}), 200
\end{lstlisting}

\begin{lstlisting}[language=JavaScript, caption={Frontend AbortController Entegrasyonu}]
// React bileşeninde Stop Response mantığı
const controllerRef = useRef(null);

const handleStop = async () => {
  // SSE bağlantısını kes
  if (controllerRef.current) {
    controllerRef.current.abort();
  }

  // Backend'i bilgilendir
  await fetch("/api/cancel", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ request_id: currentRequestId }),
  });

  setStatus("Yanıt durduruldu.");
};
\end{lstlisting}

\subsubsection{Kullanıcı Arayüzü Davranışı}

\begin{notebox}[Kullanıcı Deneyimi Kararları]
\begin{itemize}[leftmargin=1.5em, itemsep=2pt]
    \item Yanıt durdurulduğunda arayüzde \emph{"Yanıt durduruldu"} mesajı belirir.
    \item Gereksiz hata bildirimleri (pop-up) bastırılır; kullanıcı deneyimi bozulmaz.
    \item Kısmen alınan yanıt ekranda korunur; kullanıcı isterse devam isteyebilir.
    \item Token düşümü, alınan kısmın uzunluğuyla orantılı olacak şekilde planlanmaktadır (gelecek sprint).
\end{itemize}
\end{notebox}

\subsection{Güvenlik ve Ölçeklenebilirlik Değerlendirmesi}

Mevcut implementasyonda \texttt{CANCELLED\_REQUESTS} sözlüğü sunucu belleğinde tutulmaktadır. Bu yaklaşım tekli sunucu ortamlarında yeterlidir; ancak yatay ölçekleme (horizontal scaling) senaryolarında farklı sunucu örnekleri aynı sözlüğe erişemeyeceği için sorun yaratabilir.

\begin{warnbox}[Teknik Borç: Redis Entegrasyonu]
Yük dengelemeli (load-balanced) üretim ortamına geçişte \texttt{CANCELLED\_REQUESTS} sözlüğünün \textbf{Redis} gibi merkezi bir anahtar-değer deposuyla değiştirilmesi gerekmektedir. Bu geçiş, mevcut arayüzün değiştirilmesini gerektirmez; yalnızca \texttt{is\_request\_cancelled} ve \texttt{cancel\_request} fonksiyonlarının Redis istemcisiyle güncellenmesi yeterlidir.
\end{warnbox}

\section{Model Kararlılığı ve Agent Yapılandırması}

\subsection{Embedding Modeli Optimizasyonu}

Sistemde kullanılan semantik arama (RAG) altyapısının kararlılığını artırmak amacıyla embedding model hiyerarşisi modernize edilmiştir.

\begin{warnbox}[404 Not Found Hatalarının Giderilmesi]
Önceki konfigürasyonda yer alan \texttt{text-embedding-004} ve \texttt{005} modellerinin deprecated olması sebebiyle yaşanan gereksiz retry (yeniden deneme) döngüleri kaldırılmıştır. Sistem doğrudan \texttt{gemini-embedding-001} modeline yönlendirilerek gecikme süreleri düşürülmüştür.
\end{warnbox}

\subsection{Gemini 3 Flash ve Agent Mode Desteği}

Agent Mode (Ajan Modu) kullanımında karşılaşılan "boş yanıt" sorunu, modelin akıl yürütme (reasoning) yeteneklerini tetikleyen \texttt{thinking\_config} parametresinin eksikliğinden kaynaklanmaktaydı.

\begin{itemize}[leftmargin=2em]
    \item \textbf{Thinking Config Entegrasyonu:} Gemini 3 modelleri için \texttt{thinking\_level: "LOW"} parametresi \texttt{GeminiAdapter} katmanına entegre edilmiştir.
    \item \textbf{SDK Uyumluluğu:} \texttt{google-genai} SDK versiyon farklılıklarına karşı hem tip güvenli (\texttt{ThinkingConfig}) hem de sözlük tabanlı (fallback) yapı oluşturulmuştur.
\end{itemize}

\begin{notebox}[Model Modernizasyonu]
Platformun varsayılan modeli \texttt{gemini-1.5-flash} sürümünden daha güncel ve performanslı olan \texttt{gemini-2.5-flash} sürümüne yükseltilmiştir.
\end{notebox}

\subsection{Öğrenilen Teknik Kavramlar}

Bu hafta öne çıkan teknik öğrenimler şunlardır:

\begin{itemize}[leftmargin=2em, itemsep=4pt]
    \item \textbf{Python \texttt{try...finally}:} İstisna durumlarından bağımsız olarak kritik işlemlerin (billing, kaynak temizliği) yürütülmesini garanti eden yapı.
    \item \textbf{Server-Sent Events (SSE):} Sunucudan istemciye tek yönlü, düşük gecikmeli veri akışı sağlayan HTTP tabanlı protokol.
    \item \textbf{Hibrit RAG (BM25 + Embedding):} Hem kelime eşleşmesi hem de anlamsal benzerlik üzerinden belge erişimi sağlayan çift kanallı arama mimarisi.
    \item \textbf{AbortController API:} Tarayıcı ortamında devam eden HTTP isteklerini programatik olarak iptal etmeye yarayan Web API'si.
    \item \textbf{Atomik Faturalama Semantiği:} Faturalama işleminin, bağımlı olduğu yan etkilerden (veri kaydetme) bağımsızlaştırılarak güvenilir hale getirilmesi prensibi.
    \item \textbf{Thinking Config:} Gemini 3 modellerinin "reasoning" (akıl yürütme) katmanını kontrol eden ve boş yanıt sorunlarını gideren SDK konfigürasyonu.
\end{itemize}
