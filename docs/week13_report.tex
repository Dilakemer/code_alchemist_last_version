\subsection{2.2 Temel Sorunlar Tanımlanmıştır}

\textbf{Sorun 1: Gemini Free Tier Kota Sınırı}
\begin{itemize}
    \item Free tier: Günde maksimum 20 istek
    \item Test başında 20+ istek yapılmış olduğu için quota dolu
    \item Her `/api/ask` isteği fallback zincirini tetikliyor
    \item Fallback zinciri: gemini-2.5-flash-lite → gemini-3.1-flash-lite → gemini-2.5-flash
    \item Kaynaklar: \url{https://ai.google.dev/gemini-api/docs/rate-limits}
\end{itemize}

\textbf{Sorun 2: Sıralı (Sequential) Model Zinciri}
\begin{lstlisting}[language=Python]

\subsection{6.3 Optimizasyon 3: Uvicorn Workers}

\textbf{Process-Level Paralelizm:}

\begin{lstlisting}[language=bash]
# Single Worker (ÖNCESI)
$ uvicorn backend.app_factory:create_app --factory --port 5001
INFO: Started server process [7644]

# İşletim Sistemi Perspektifi:
# 1 Python işlemi × 1 thread = 1 request işleme kapasitesi
# 5 eşzamanlı istek = Sıralı işleme = 5× daha yavaş

# Multiple Workers (SONRASI)
$ uvicorn backend.app_factory:create_app --factory \
  --port 5001 --workers 4
INFO: Started parent process [20592]
INFO: Started server process [20593]  # Worker 1
INFO: Started server process [20594]  # Worker 2
INFO: Started server process [20595]  # Worker 3
INFO: Started server process [20596]  # Worker 4

# İşletim Sistemi Perspektifi:
# 4 Python işlemi × 1 thread = 4 request işleme kapasitesi
# 5 eşzamanlı istek = 4 parallel + 1 sıra = 1.25× ölçeklendirme
\end{lstlisting}

\textbf{WsgiToAsgi Bridge Analizi:}
\begin{lstlisting}[language=Python]
# app_factory.py - Flask-Async Integration
def create_app() -> FastAPI:
    app = FastAPI()
    
    # Lifespan: Thread pool executor ayarı
    async def lifespan(app):
        # ASYNC_EXECUTOR_WORKERS = 96 (default)
        executor = ThreadPoolExecutor(max_workers=96)
        asyncio.get_event_loop().set_default_executor(executor)
        yield
    
    # Flask WSGI uygulamasını FastAPI'ye bağla
    app = WsgiToAsgi(flask_app)  # Sync → Async dönüş
\end{lstlisting}

\textbf{İyileştirme Mekanizması:}
\begin{enumerate}
    \item 4 Uvicorn Worker = 4 ayrı Python işlemi
    \item Her worker kendi async event loop'u var
    \item Her loop 96-worker thread pool'u var
    \item Flask çağrıları thread pool'da çalışıyor
    \item Sonuç: 4 worker × 96 threads = Çok daha fazla paralelizm
\end{enumerate}

\subsection{7.3 Load Testing Best Practices}

\textbf{Doğru Approach:}
\begin{enumerate}
    \item \textbf{Baseline Test:} Sorunları belirle
    \item \textbf{Bottle Neck Analizi:} Backend loglarını oku
    \item \textbf{Hypothesize:} Root cause belirle
    \item \textbf{Optimize:} Bir kerede bir değişiklik yap
    \item \textbf{Measure:} Etkisini doğrula
    \item \textbf{Repeat:} Sonraki sorun için tekrar et
\end{enumerate}

\textbf{Yaptığımız Yaklaşım:}
\begin{enumerate}
    \item ✓ Baseline: \%55 hata oranı
    \item ✓ Loglar: 429 quota error, 503 unavailable
    \item ✓ Hypothesis: Gemini quota, sequential blend, 1 worker
    \item ✓ Optimize: Model switch, blend disable, workers=4
    \item ✓ Measure: \%0 hata oranı
\end{enumerate}

\chapter{Yeni Özellik: Context Health Advisory Sistemi}

\section{Giriş ve Motivasyon}

Kullanıcılar uzun süreli sohbetlerde bağlamın dolması nedeniyle performans düşüşü veya token aşımı hataları (örneğin 429 veya 503) yaşıyorlardı. Ayrıca mevcut konuşmanın konusu değiştiğinde LLM'in önceki bağlamdan etkilenerek yanlış kararlar alması bir problemdi. Bu sorunları kullanıcıyı sıkmadan (non-intrusive) çözmek amacıyla \textbf{Bağlam Sağlığı Uyarı Sistemi (Context Health Advisory System)} geliştirildi.

\section{Sistem Mimarisi}

Sistem üç ana bileşenden oluşmaktadır:
\begin{enumerate}
    \item \textbf{Backend Health Analyzer:} Token baskısını ve konu kaymasını (topic drift) hesaplayan analiz motoru.
    \item \textbf{SSE Injection Layer:} Hem normal model akışında hem de Agent Mode akışında stream bitmeden hemen önce çalışan enjeksiyon katmanı.
    \item \textbf{Frontend UI Component:} Kullanıcı deneyimini bozmayan, kapatılabilir (dismissible) ve cooldown mekanizmasına sahip bilgilendirme afişi (Advisory Banner).
\end{enumerate}

\subsection{1. Backend: ContextHealthAnalyzer}

\texttt{server/backend/runtime/limits.py} içerisindeki \texttt{ContextHealthAnalyzer} sınıfı güçlendirildi.

\begin{lstlisting}[language=Python]
# Token Baskisi (Context Bloat) Hesaplama
def calculate_pressure(self) -> float:
    # prompt + rag + history char counts
    total_chars = len(sys_prompt) + len(rag) + self._history_chars()
    return min(1.0, (total_chars // 4) / max_tokens)

# Konu Kaymasi (Topic Drift) Hesaplama
def calculate_drift_score(self, intent, touched_files):
    # Jaccard Benziligi: Son 3 mesaj vs Yeni Soru
    q_words = self._keyword_set(question)
    h_words = self._keyword_set(history_text)
    overlap = len(q_words & h_words) / len(q_words | h_words)
    
    # Agirlikli hesaplama (Keyword Drift, File Drift, Intent Shift)
    return (1.0 - overlap) * 0.55 + (1.0 - file_overlap) * 0.25 + ...
\end{lstlisting}

\subsection{2. Backend: SSE Stream Integration}

\texttt{app.py} içerisine normal modeller için non-blocking bir yardımcı fonksiyon eklendi ve SSE (Server-Sent Events) jeneratörüne entegre edildi.

\begin{lstlisting}[language=Python]
def _check_context_health(history, question, conv_id):
    # Cooldown mekanizmasi: Ayni sohbette ust uste cikmasini engeller
    if (history_len - last_at) < _ADVISORY_MSG_COOLDOWN:
        return None

    if total_chars >= _ADVISORY_BLOAT_CHARS:

\end{lstlisting}

\subsection{3. Frontend: Advisory Banner}

\texttt{client/src/App.jsx} SSE parser'ina \texttt{advisory} event tipi eklendi ve state \texttt{ChatInterface.jsx} bileşenine geçirildi. Glassmorphism tasarım diliyle uyumlu \texttt{AdvisoryBanner.jsx} geliştirildi.

\section{Sonuç ve Avantajlar}

Bu sistem sayesinde:
\begin{itemize}
    \item \textbf{Token Tasarrufu:} Kullanıcılar bağlam dolmadan yeni sohbete yönlendiriliyor.
    \item \textbf{Yüksek Başarı Oranı:} LLM, konu dışı eski bağlamdan etkilenmiyor.
    \item \textbf{UX Dostu:} Modal (popup) yerine ekranı bloke etmeyen "sticky banner" kullanıldı ve \texttt{ADVISORY\_MSG\_COOLDOWN} (varsayılan: 10 mesaj) sayesinde kullanıcı spam'den korundu.
\end{itemize}

\chapter{Yeni Platform: Mobil Uygulama Geliştirme Sürecinin Başlaması}

\section{Giriş}

Projenin 13. haftasında, Code Alchemist platformunun erişilebilirliğini artırmak ve kullanıcılara mobil cihazlarda da AI destekli kod asistanı deneyimini sunmak amacıyla mobil uygulama geliştirme sürecine başlanmıştır. Mobil uygulama, React Native ve Expo altyapısı kullanılarak inşa edilmeye başlanmış olup, backend ile tam uyumlu çalışacak Service Oriented Architecture (SOA) mimarisi temel alınmıştır.

\section{Mobil Uygulama Mimarisi ve SOA Yaklaşımı}

Mobil projenin kod tabanı (codebase) modüler ve ölçeklenebilir bir mimari hedefiyle tasarlanmıştır. Uygulamanın sunucu (backend) ile olan tüm iletişimi soyutlanmış ve merkezi bir servis katmanı üzerinden yönetilmektedir.

\subsection{1. Servis Katmanı (Service Layer) Entegrasyonu}

Tüm API istekleri \texttt{src/services/api.js} dosyası üzerinden yürütülmektedir. Bu dosya, backend'in sunduğu tüm endpoint'leri sarmalayarak (wrapper) UI bileşenlerinden API karmaşıklığını gizlemektedir.

\begin{lstlisting}[language=JavaScript]
// src/services/api.js - Merkezi API İletişim Servisi
import { API_BASE, CLIENT_SOURCE } from '../config';

const request = async (path, { method = 'GET', token, body, headers = {} } = {}) => {
  // ... Timeout ve AbortController yönetimi ...
  const response = await fetch(`${API_BASE}${path}`, {
    method,
    headers: {
      ...(body ? { 'Content-Type': 'application/json' } : {}),
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
      'X-Client-Source': CLIENT_SOURCE,
      ...headers,
    },
    ...(body ? { body: JSON.stringify(body) } : {}),
    signal: controller.signal,
  });
  // ... Error handling ve Payload parse işlemleri ...
  return data;
};

// Modüler Endpoint Dışa Aktarımları
export const askQuestion = ({ token, question, code = '', model = 'auto', conversationId = null }) =>
  request('/api/ask', { method: 'POST', token, body: { question, code, model, conversation_id: conversationId }});

export const login = ({ email, password }) =>
  request('/api/auth/login', { method: 'POST', body: { email, password } });
\end{lstlisting}

\subsection{2. Durum ve Depolama Yönetimi (Storage Layer)}

Kullanıcı oturum yönetimi ve yetkilendirme token'ları, AsyncStorage kullanılarak \texttt{src/services/storage.js} üzerinden izole edilmiştir. SOA mimarisinin getirdiği sorumlulukların ayrılması (Separation of Concerns) prensibi burada da uygulanmıştır.

\begin{lstlisting}[language=JavaScript]
// src/services/storage.js
import AsyncStorage from '@react-native-async-storage/async-storage';
import { STORAGE_KEYS } from '../config';

export const saveSession = async ({ token, user }) => {
  await AsyncStorage.multiSet([
    [STORAGE_KEYS.token, token || ''],
    [STORAGE_KEYS.user, user ? JSON.stringify(user) : ''],
  ]);
};
\end{lstlisting}

\subsection{3. Gerçek Zamanlı İletişim (Real-time Collaboration)}

Web uygulamasındaki ortak çalışma (collaboration) yeteneğinin mobile taşınması amacıyla \texttt{Socket.io} istemcisi entegre edilmiş ve \texttt{src/hooks/useCollabSocket.js} custom hook'u olarak soyutlanmıştır. Bu sayede UI katmanı, socket bağlantı yönetimi detaylarından arındırılmıştır.

\begin{lstlisting}[language=JavaScript]
// src/hooks/useCollabSocket.js
export function useCollabSocket(token, userToken, userName = 'Guest', onHistoryRefresh = null) {
  // ... Socket state'leri ...
  useEffect(() => {
    const socket = io(SOCKET_BASE, {
      auth: { token: userToken },
      transports: ['websocket', 'polling']
    });

    socket.on('collab_stream_chunk', (data) => {
      if (data.token !== token) return;
      setLiveStreamText((prev) => prev + data.chunk);
    });
    // ... Socket olay dinleyicileri ve temizleme ...
  }, [token, userToken]);
  
  return { connected, liveStreamText, isStreaming };
}
\end{lstlisting}

\section{Ana Uygulama (App.js) ve Yönlendirme (Routing)}

Mobil uygulamanın giriş noktası olan \texttt{App.js}, durumu (state) merkezileştiren ve yetkilendirme akışını yöneten bir orkestratör görevi görmektedir. Kapsamlı bir React Navigation kütüphanesi yerine, başlangıç aşamasında basit ve etkili bir custom view routing (\texttt{activeView}) mekanizması tercih edilmiştir.
