\section{Özellik 1: Bağlam Optimizasyon Asistanı}

\subsection{Ürün Gereksinimi}
Prompt göndermeden önce, kaliteyi koruyarak gereksiz token yükünü azaltacak öneriler sunulması. Auto mode açık olduğunda sistemin bunu akış içinde otomatik uygulayabilmesi.

\subsection{Uygulama Günlüğü — Ne Eklendi?}

\textbf{Dosya:} \texttt{client/src/components/ChatInterface.jsx}

\textbf{Eklenen Yardımcı İşlevler:}
\begin{itemize}[leftmargin=1.5em]
  \item Tekrarlı satır temizleme
  \item Gereksiz boşlukları sıkıştırma
  \item Token öncesi/sonrası kestirimi
\end{itemize}

\textbf{Eklenen UI bileşenleri:}
\begin{itemize}[leftmargin=1.5em]
  \item Auto Optimize: ON/OFF anahtarı
  \item Yüksek bağlam baskısında gönderim öncesi öneri kartı
  \item Üç aksiyon butonu: \textit{Apply Optimized Text}, \textit{Send Optimized}, \textit{Send Original}
\end{itemize}

\subsection{Davranış Akışı}

\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Adım 1.} Kullanıcı prompt yazar.
  \item \textbf{Adım 2.} Auto optimize açıksa rapor otomatik oluşturulur; kapalıysa kullanıcı manuel olarak başlatabilir.
  \item \textbf{Adım 3.} Kullanıcı üç seçenekten birini seçer: optimize metni uygula, optimize haliyle gönder veya orijinal metinle devam et.
  \item \textbf{Adım 4.} Kullanıcı, promptu göndermeden önce tahmini token tasarrufunu sayısal olarak görebilir.
\end{enumerate}

\begin{notebox}[Hafta 8 Kazanımı]
Context bar yalnızca gösteren bir widget olmaktan çıktı; aksiyon üreten bir yardımcıya dönüştü.
\end{notebox}

\subsection{Kısa Senaryo (Önce / Şimdi)}

\begin{notebox}[Gerçek Kullanım Farkı]
\textbf{Önce:} Kullanıcı 2--3 farklı kod parçası ve uzun açıklama ile prompt gönderiyor, token yükünü ancak cevap geç gelince fark ediyordu.\\[4pt]
\textbf{Şimdi:} Göndermeden hemen önce sistem tekrarlı kısımları sıkıştırıp tahmini token kazancını gösteriyor; kullanıcı optimize veya orijinal metinle devam etmeyi seçiyor.
\end{notebox}

\subsection{UI Tarafında Beklenen Davranış}

Yukarıdaki 12 durumdan biri algılandığında, optimize kartı aşağıdaki aksiyonları sunmalıdır:

\begin{notebox}[Optimize Kartı — Zorunlu Aksiyon Butonları]
\begin{itemize}[leftmargin=1.2em,nosep]
  \item \textbf{Apply Optimized Text} — Optimize metni prompt kutusuna uygular; kullanıcı göndermeden önce görebilir.
  \item \textbf{Send Optimized} — Optimize edilmiş metni doğrudan gönderir.
  \item \textbf{Send Original} — Orijinal metni değiştirmeden gönderir; öneriyi atlar.
\end{itemize}
\end{notebox}

\begin{warnbox}[Önemli UX Notu]
Kullanıcıya her zaman seçim hakkı verilmeli; sistem asla otomatik olarak orijinal promptun üzerine yazmamalıdır. Auto Optimize açık olsa bile nihai karar kullanıcıya aittir.
\end{warnbox}

\subsection{Sınırlar ve Sonraki Adımlar}
\begin{itemize}[leftmargin=1.5em]
  \item Prompt optimizasyonu şu an kural tabanlıdır; 12 kategori heuristic ile tespit edilmektedir.
  \item İleride model destekli semantic compression eklenebilir.
  \item Kategori listesi genişleyebilir; yeni prompt anti-pattern'ler tespit edildikçe tablo güncellenmelidir.
\end{itemize}

\subsubsection{Live Sync Uygulama Adımları}

\begin{enumerate}[leftmargin=1.5em]
  \item \textbf{Optimistic UI Update} — Soru hemen chat history'ye eklenir.
  \item \textbf{3-Attempt Retry Loop} — Cold-start ve geçici ağ hatalarına karşı dayanıklılık (600ms + 1200ms exponential backoff).
  \item \textbf{HTTP POST to /send} — Payload: \texttt{\{question, model, sender\_name, client\_nonce\}}.
  \item \textbf{Backend Session Oluşturma} — History kaydı yaratılır; \texttt{history\_id} döner.
  \item \textbf{Background Thread Başlatma} — Socket.io emit için background thread başlar.
  \item \textbf{Stream Token Akışı} — Backend, seçilen modele göre stream başlar:\\
        — Gemini: \texttt{generate\_content(stream=True)} ile iter.\\
        — GPT: OpenAI SDK ile stream completions.\\
        — Claude: Anthropic SDK ile \texttt{messages.stream} loop.
  \item \textbf{Socket Emit (Per Chunk)} — Her chunk, Socket.io ile room'a broadcast edilir: \texttt{collab\_stream\_chunk}.
  \item \textbf{Frontend Socket Listener} — \texttt{useCollabSocket} hook, chunk'ları local state'e toplar.
  \item \textbf{Stream Tamamlanma} — \texttt{collab\_stream\_done} event ile full response gönderilir.
  \item \textbf{Database Persist} — Backend, stream tamamlandıktan sonra full response'u commit eder.
\end{enumerate}

  \item Tüm mevcut call sites değiştirilmeden korundu; backward-compatible.
\end{itemize}
\end{notebox}

\textbf{Değişen dosyalar:}
\begin{itemize}[leftmargin=1.5em]

\subsubsection{2. Generator Response Type Hatası Düzeltmesi}

\textbf{Problem:} \texttt{'generator' object has no attribute 'text'} — Compat wrapper'daki \texttt{yield} statement, non-stream modunda da generator döndürüyordu.

\textbf{Çözüm:} Stream iterator'ü inner generator function'a wrap edildi:

\begin{lstlisting}[language=Python, caption={Düzeltilmiş Kod (app.py:80--91)}]
def generate_content(self, prompt, stream=False):
    if stream:
        def _iter_stream():
            for chunk in self.client.models.generate_content(
                prompt, stream=True):
                yield chunk
        return _iter_stream()
    else:
        # Non-stream: normal response, yield yok
        return self.client.models.generate_content(
            prompt, stream=False)
\end{lstlisting}

\subsubsection{3. Transient Network Resilience (Retry Loop)}

\textbf{Problem:} Render cold-start ve DNS gecikmeleri sırasında collaboration mesaj gönderimi başarısız oluyordu.

\textbf{Çözüm:} 3-attempt retry loop ile exponential backoff (600ms → 1200ms):

\begin{lstlisting}[language=Java, caption={App.jsx:857--885}]
const collabUrl =
    /api/collaboration/session/${collabToken}/send;
for (let attempt = 0; attempt < 3; attempt += 1) {
    try {
        resp = await fetch(collabUrl, { method: 'POST', ...opts });
        break; // Basarili
    } catch (err) {
        lastErr = err;
        if (attempt < 2) {
            // attempt=0: 600ms, attempt=1: 1200ms
            await new Promise(resolve =>
                setTimeout(resolve, 600 * (attempt + 1)));
        }
    }
}
\end{lstlisting}

\subsubsection{4. Global Exception Handler HTTP Status Koruması}

\textbf{Problem:} Flask global exception handler tüm hataları 500'e dönüştürüyordu; 404 hataları da 500 olarak görünüyordu.

\textbf{Çözüm:} HTTPException type check eklendi:

\begin{lstlisting}[language=Python, caption={app.py:532--547}]
@app.errorhandler(Exception)
def handle_exception(e):
    from werkzeug.exceptions import HTTPException
    if isinstance(e, HTTPException):
        return jsonify({'error': str(e.description)}), e.code
    return jsonify({'error': 'Internal server error'}), 500
\end{lstlisting}

\subsubsection{5. NPM Güvenlik Açıkları Giderildi}

\textbf{Problem:} \texttt{npm audit} 10 vulnerability tespit etti (1 critical jsPDF, 4 high, 5 moderate).

\textbf{Çözüm:}
\begin{itemize}[leftmargin=1.5em]
  \item \texttt{npm audit fix} ile otomatik düzeltilebilir açıklar kapandı.
  \item Forced upgrade: \texttt{jspdf@\^{}4.2.1} (critical LFI/DoS düzeltmesi).
  \item Override: \texttt{dompurify@\^{}3.3.3} (XSS mitigation).
\end{itemize}

\textbf{Sonuç:} 0 vulnerability — production deployment güvenli.

\subsection{Sınırlar ve Sonraki Adımlar}
\begin{itemize}[leftmargin=1.5em]
  \item Backend'de eventlet worker kullanılmaktadır; Render.com'da WebSocket veya Polling transport otomatik seçilir.
  \item Collaboration endpoint'leri şu an sahip doğrulaması (ownership check) yapmamaktadır; token'ı bilen herkes yorum ekleyebilir ve durum değiştirebilir. Sonraki iterasyonda reviewer kimlik doğrulaması eklenmesi önerilir.
\end{itemize}
