\section{Özellik 1: Bağlam Optimizasyon Asistanı}
% ═══════════════════════════════════════════════════════════════════════════════

\subsection{Ürün Gereksinimi}

Prompt göndermeden önce, kaliteyi koruyarak gereksiz token yükünü azaltacak öneriler sunulması.
Auto mode açık olduğunda sistemin bunu akış içinde otomatik uygulayabilmesi.

\subsection{Uygulama Günlüğü — Ne Eklendi?}

\textbf{Dosya:} \texttt{client/src/components/ChatInterface.jsx}

\begin{tcolorbox}[codebox, title={Eklenen Yardımcı İşlevler}]

  \item Tekrarlı satır temizleme
  \item Gereksiz boşlukları sıkıştırma
  \item Token öncesi/sonrası kestirimi
\end{itemize}
\end{tcolorbox}

\textbf{Eklenen UI bileşenleri:}
\begin{itemize}[leftmargin=*, label=\textcolor{secondaryblue}{\textbullet}]

  \item Yüksek bağlam baskısında gönderim öncesi öneri kartı
  \item Üç aksiyon butonu: \texttt{Apply Optimized Text}, \texttt{Send Optimized}, \texttt{Send Original}

\subsection{Davranış Akışı}

\begin{enumerate}[leftmargin=*, label=\textbf{Adım \arabic{enumi}.}]
  \item Kullanıcı prompt yazar.
  \item Auto optimize açıksa rapor oluşturulur; kapalıysa kullanıcı manuel olarak başlatabilir.
  \item Kullanıcı üç seçenekten birini seçer:
        optimize metni uygula, optimize haliyle gönder veya orijinal metinle devam et.
  \item Kullanıcı, promptu göndermeden önce token tasarrufunu sayısal olarak görebilir.
\end{enumerate}

\subsection{Kazanım}

\begin{tcolorbox}[successbox, title={Hafta 8 Kazanımı}]
Context bar yalnızca gösteren bir widget olmaktan çıktı; aksiyon üreten bir yardımcıya dönüştü.
\end{tcolorbox}

\subsection{Kısa Senaryo (Önce / Şimdi)}

\begin{tcolorbox}[infobox, title={Gerçek Kullanım Farkı}]
\begin{itemize}[leftmargin=*, nosep, label=\textbullet]
  \item \textbf{Önce:} Kullanıcı 2--3 farklı kod parçası ve uzun açıklama ile prompt gönderiyor,
        token yükünü ancak cevap geç gelince fark ediyordu.
  \item \textbf{Şimdi:} Göndermeden hemen önce sistem tekrarlı kısımları sıkıştırıp
        tahmini token kazancını gösteriyor; kullanıcı optimize veya orijinal metinle devam etmeyi seçiyor.
\end{itemize}
\end{tcolorbox}

\subsection{Tekrar Dışı Optimizasyon Önerileri ve Örnek Hatalı Promptlar}

Aşağıdaki maddeler, sadece tekrar tespiti dışında kalan bağlam optimizasyon alanlarıdır.
Her madde için bir \textit{hatalı prompt} örneği verilmiş; sistemin önermesi beklenen iyileştirme de belirtilmiştir.

\vspace{4pt}
\begin{longtable}{@{} C{0.55cm} P{3.5cm} P{5.0cm} P{4.5cm} @{}}
\toprule
\textbf{No} & \textbf{Optimizasyon Fikri} & \textbf{Örnek Hatalı Prompt} & \textbf{Beklenen Önerme} \\
\midrule
\endhead
1 & Gereksiz bağlam budama &
  \textit{"Bugün kahve içtim, toplantı yaptım, sonra markete gittim, bu arada kodu da düzelt."} &
  Teknik hedefi tut, hikaye/gürültüyü kaldır \\[5pt]
2 & Amaç netleştirme &
  \textit{"Bunu hızlandır, güvenli yap, UI tasarla, deploy et, test yaz."} &
  Tek ana hedef seç ve önceliklendir \\[5pt]
3 & Çıktı formatı standardizasyonu &
  \textit{"Nasıl uygun görürsen öyle cevapla."} &
  Tablo / madde / kod bloğu gibi net format belirt \\[5pt]
4 & Kısıt ekleme &
  \textit{"Bu problemi çözer misin?"} &
  Dil, uzunluk, kapsam, karmaşıklık sınırı ekle \\[5pt]
5 & Rol ve hedef kitle uyumu &
  \textit{"Bu kodu anlat."} &
  Kime anlatıldığını belirt (junior / senior / PM) \\[5pt]
6 & Kod sinyali güçlendirme &
  \textit{"Aşağıdakini iyileştir: def f(x): return x*x"} &
  Dil ve istenen çıktı tipini netleştir (diff / fonksiyon) \\[5pt]
7 & Test odaklı tamamlama &
  \textit{"Bu fonksiyonu düzelt."} &
  Edge-case ve basit test maddeleri ekle \\[5pt]
8 & Risk ve güvenlik filtresi &
  \textit{"Auth token akışını komple değiştir, direkt patch ver."} &
  Risk analizi + rollback planı ile ilerle \\[5pt]
9 & Token maliyeti azaltma &
  \textit{"1200 satır logu komple analiz et."} &
  Yalnız ilgili log kesitlerini iste \\[5pt]
10 & Çok adımlı görev ayrışımı &
  \textit{"Bu ürünü baştan sona optimize et."} &
  Analiz / çözüm / test adımlarına böl \\[5pt]
11 & Belirsizlik tespiti &
  \textit{"Çalışmıyor, düzelt."} &
  Hata mesajı, tekrar adımı, ortam bilgisi iste \\[5pt]
12 & Dil kalitesi sadeleme &
  \textit{"Aynı şeyi tekrar ediyorum, iyi olsun, daha iyi olsun."} &
  Tekrarı azalt, net ve kısa hedef cümlesi üret \\
\bottomrule
\end{longtable}

\subsection{UI Tarafında Beklenen Davranış (Özellik 1 için)}

Yukarıdaki 12 durumdan biri algılandığında, optimize kartı aşağıdaki aksiyonlardan en az birini sunmalıdır:

\begin{tcolorbox}[codebox, title={Optimize Kartı — Zorunlu Aksiyon Butonları}]
\begin{itemize}[leftmargin=*, nosep, label=\textcolor{primaryblue}{\textbullet}]
  \item \texttt{Apply Optimized Text} — Optimize metni prompt kutusuna uygular, kullanıcı göndermeden önce görebilir.
  \item \texttt{Send Optimized} — Optimize edilmiş metni doğrudan gönderir.
  \item \texttt{Send Original} — Orijinal metni değiştirmeden gönderir; önermeyi atlar.
\end{itemize}
\end{tcolorbox}

\begin{tcolorbox}[warningbox, title={Önemli UX Notu}]
Kullanıcıya \textbf{her zaman seçim hakkı} verilmeli; sistem asla otomatik olarak
orijinal promptun üzerine yazmamalıdır. Auto Optimize açık olsa bile nihai karar kullanıcıya aittir.
\end{tcolorbox}

\subsection{Sınırlar ve Sonraki Adımlar}

\begin{itemize}[leftmargin=*, label=\textcolor{warningorange}{\textbullet}]
  \item Prompt optimizasyonu şu an kural tabanlıdır; 12 kategori heuristic ile tespit edilmektedir.
  \item İleride model destekli \textit{semantic compression} eklenebilir.
  \item Kategori listesi genişleyebilir; yeni prompt anti-pattern'ler tespit edildikçe tablo güncellenmelidir.
\end{itemize}

\section{Özellik 2: Patch Güvenlik Seviyesi}

\subsection{Ürün Gereksinimi}

Patch uygulamadan önce risk skorlaması yapılarak 3 modda karar verilmesi:
\textbf{güvenli uygula}, \textbf{incele}, \textbf{blokla}.

\subsection{Uygulama Günlüğü — Ne Eklendi?}

\textbf{Dosya:} \texttt{client/src/components/ChatInterface.jsx}

\textbf{Eklenen bileşenler:}
\begin{itemize}[leftmargin=*, label=\textcolor{secondaryblue}{\textbullet}]
  \item Patch risk skoru hesabı (0--100 aralığı)
  \item \textbf{Patch Safety Modal:} risk gerekçeleri listesi, review akışı için kontrollü devam,
        block durumunda uygulamayı durdurma
\end{itemize}

\textbf{Ek düzeltme:}
\begin{tcolorbox}[warningbox, title={Önemli Refactor}]

tüm patch akışı risk kapısından geçirildi.
\end{tcolorbox}

\subsection{Risk Skoru Hesaplama Örneği}

Aşağıda, aynı patch senaryosunun üç farklı profilde nasıl skorlandığı gösterilmektedir.

\subsubsection{2. Generator Response Type Bug Fix}

\textbf{Problem:} Collaboration mesaj gönderimi sırasında "Hata: 'generator' object has no attribute 'text'" hatası ortaya çıkıyordu.

\textbf{Root Cause:} Compat wrapper'daki \texttt{generate\_content()} metodunda \texttt{yield} statement'i non-stream modunda da generator döndürüyordu:

\begin{lstlisting}[language=Python, caption={Eski (Hatalı) Kod}]
def generate_content(self, prompt, stream=False):
    # HATA: yield statement'i HER YOLDA generator'ı tetikliyor
    for chunk in self.client.models.generate_content(prompt, stream=True):
        yield chunk
    
    # Non-stream istekler de generator döner → property access hataları
\end{lstlisting}

\textbf{Çözüm:} Stream iterator'ü inner generator function'a wrap edildi:

\begin{lstlisting}[language=Python, caption={Yeni (Sabit) Kod}]
def generate_content(self, prompt, stream=False):
    if stream:
        def _iter_stream():
            for chunk in self.client.models.generate_content(prompt, stream=True):
                yield chunk
        return _iter_stream()
    else:
        # Non-stream: normal response döner, yield yok
        return self.client.models.generate_content(prompt, stream=False)
\end{lstlisting}

\textbf{Dosya:} \texttt{server/app.py}, satırlar 80--91.

\textbf{Sonuç:} ✅ Collaboration mesajları başarıyla streaming; response attribute access hataları çözüldü.

\subsubsection{3. Transient Network Resilience (Retry Loop)}

\textbf{Problem:} Render cold-start ve DNS gecikmeleri sırasında collaboration mesaj gönderimi başarısız oluyordu; kullanıcı "Load failed" hatasıyla karşılaşıyordu.

\textbf{Çözüm:} Frontend'de 3-attempt retry loop with exponential backoff (600ms, 1200ms):

\begin{lstlisting}[language=JavaScript, caption={client/src/App.jsx Satırlar 857--885}]
const collabUrl = `/api/collaboration/session/${collabToken}/send`;
for (let attempt = 0; attempt < 3; attempt += 1) {
  try {
    resp = await fetch(collabUrl, { method: 'POST', ... });
    break; // Success
  } catch (err) {
    lastErr = err;
    if (attempt < 2) {
      // 600ms (attempt 1), 1200ms (attempt 2)
      await new Promise(resolve => 
        setTimeout(resolve, 600 * (attempt + 1))
      );
    }
  }
}
\end{lstlisting}

\textbf{Sonuç:} ✅ Transient ağ hataları otomatik kurtarılır; UX düzeltilmiş.

\subsubsection{4. Global Exception Handler HTTP Status Preservation}

\textbf{Problem:} Flask global exception handler, tüm HTTP errors'ları 500'e dönüştürüyordu; 404 hataları ürün tarafında 500 olarak görünüyordu.

\textbf{Çözüm:} Exception handler'a HTTPException check'i eklendi (app.py satırlar 532--547):

\begin{lstlisting}[language=Python]
@app.errorhandler(Exception)
def handle_exception(e):
    from werkzeug.exceptions import HTTPException
    
    if isinstance(e, HTTPException):
        # Preserve orginal status code
        return jsonify({'error': str(e.description)}), e.code
    else:
        # Only non-HTTP exceptions become 500
        return jsonify({'error': 'Internal server error'}), 500
\end{lstlisting}

\textbf{Sonuç:} ✅ HTTP 404/405 properly returned; error logging accurate.
